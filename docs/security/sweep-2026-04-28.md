## 2. Detalhamento das 13 vulns npm

Todas vêm via `devDependencies`. **Nenhuma vai pro `dist/` final**.

### High (6)

| Pacote | Versão afetada | Advisory | Caminho |
|---|---|---|---|
| `minimatch` | 9.0.0–9.0.6 | GHSA-3ppc-4f35-3m26 (ReDoS via wildcards), GHSA-7r86-cg39-jmmj, GHSA-23c5-xmqv-rm74 | `@typescript-eslint/eslint-plugin → minimatch` (lint-time only) |
| `@typescript-eslint/typescript-estree` | 6.16.0–7.5.0 | Depende de minimatch acima | lint-time |
| `@typescript-eslint/parser` | 6.16.0–7.5.0 | Depende de typescript-estree | lint-time |
| `@typescript-eslint/utils` | (transitive) | Idem | lint-time |
| `@typescript-eslint/type-utils` | (transitive) | Idem | lint-time |
| `@typescript-eslint/eslint-plugin` | (top-level) | Idem | lint-time |

**Impacto real**: ReDoS no `minimatch` só seria explorável se atacante controlasse os patterns que o eslint processa. Em projeto fechado (nosso caso), os patterns vêm de configs locais — não há vetor.

### Moderate (7)

| Pacote | Advisory | Caminho |
|---|---|---|
| `esbuild` ≤0.24.2 | GHSA-67mh-4wv8-2f99 | `vite → esbuild` (dev server only) |
| `vite` ≤6.4.1 | Transitive | dev server |
| `vite-node` | Transitive | test runner |
| `@vitest/mocker` | Transitive | test runner |
| `vitest` 0.0.1–3.0.0-beta.4 | Transitive | test runner |
| `@vitest/coverage-istanbul` | Transitive | test runner |
| `@vitest/coverage-v8` | Transitive | test runner |

**Impacto real**: GHSA-67mh-4wv8-2f99 do esbuild permite que sites externos enviem requests pro Vite dev server (CORS bypass durante desenvolvimento). Ataque requer:
1. Dev rodando `npm run dev` localmente
2. Dev visita site malicioso no mesmo browser
3. Site malicioso envia requests pro `localhost:5176`

**Mitigação atual**: Vite dev server roda em `127.0.0.1` (não `0.0.0.0`) — confirmado em `ss -tlnp`. Site externo não consegue alcançar `127.0.0.1` do browser do dev.

**Em prod**: Vite dev server não existe. Bundle estático servido via CDN.

### Por que não fizer fix agora

`npm audit fix` (sem --force) reporta zero auto-fixes possíveis (deps são todas major bumps).

`npm audit fix --force` faria upgrade de:
- vite 5 → 8 (breaking — config muda)
- vitest 2 → 4 (breaking — API muda, testes podem quebrar)
- @typescript-eslint 7 → 9 (breaking — rules mudam)

**Custo**: 4–8h de migração + risco de regressão nos 392 tests passando.
**Benefício real**: zero — todas as vulns são dev-only.

**Recomendação**: deixar dependabot fazer upgrade gradual (já configurado em `.github/dependabot.yml`). Auto-merge quando minor/patch. Major requer revisão.

### Lições aprendidas das mudanças do checkout flow

Os 3 commits recentes (`f032d91`, `97a4738`, `9e489bf`) introduziram:

1. **`fetchWithAutoRefresh`** em api.ts — bem desenhado:
   - Lock `inflightRefresh` previne refresh paralelo (race)
   - `clearSession()` limpa user state em refresh failure
   - Tenta refresh **uma vez** apenas (não loop infinito)

2. **`paymentResultFromApi`** exposto pra OrderConfirmation reusar. Não introduz superfície nova — é mesmo parsing já existente.

3. **redact.go**: removeu `number` genérico, manteve `cardnumber`/`pannumber`/`cvv`/`cvc`/`cpf`/`cnpj` específicos. Boleto `number` (linha digitável) agora visível, que é correto (é dado público pra pagamento).

Verificado linha-a-linha — sem regressão de PII protection.
